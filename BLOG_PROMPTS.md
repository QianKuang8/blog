# 博客写作 Prompts

本文件用于从 `sources/orig/<slug>.md` 撰写和检查博客文章。当前流程不要求 NotebookLM 或 `nlm` 摘要材料。

## 写作 Prompt

你要基于 `sources/orig/<slug>.md` 写一篇中文博客解读，不要写成逐段摘要、资料卡片或提纲式笔记。

目标读者是有 AI 工程背景、关注 coding agent / context engineering / LLM 产品工程的读者。基础概念不需要科普，但文章中特有的机制、术语和设计取舍必须解释清楚。

文章必须完成以下理解动作，但小节标题可以根据具体文章重写，不要机械使用模板标题：

- 开头先给判断：这篇文章为什么值得读，它真正讨论的问题是什么。
- 定义原文的核心问题。
- 解释核心机制、方案或关键论点。
- 写出重要 trade-off、限制条件或隐含前提。
- 给出明确但克制的个人判断。
- 结尾保留原文链接。

写作要求：

- 默认写成 1200-2000 字左右的中文博客解读。
- 机制复杂、值得深挖时，可以写到 2000-3500 字。
- 原文本身特别长、信息量高，且确实需要系统解读时，可以超过 3500 字，但不能只是复述原文。
- 不追求覆盖原文全部细节，优先覆盖核心主张、关键机制、重要例子、trade-off 和工程启发。
- 重要判断必须由原文中的具体信息支撑，例如机制名、系统设计、实验结果、产品行为、术语、案例或数字。
- 需要在关键判断处区分原文观点和个人判断。可以使用“文章认为”“作者的核心判断是”“我的理解是”“我觉得更重要的是”等自然表达；不要机械地把每节拆成“原文观点 / 我的评论”。
- 可以少量引入通用背景知识或本博客已有主题脉络，但不要让外部背景盖过原文。不要引入未经确认的具体事实、数据或新闻。
- 可以引用原文中的关键句、定义、术语、数据或判断；引用数量不做硬性限制，但每处引用都必须服务于文章论点。引用后应有中文转述或分析，说明它为什么重要。
- 如果引用英文原文中的关键定义或判断，可以保留英文原句，并紧接中文解释；普通内容优先中文转述。
- 文章使用中文写作。常见技术术语可以保留英文，如 coding agent、benchmark、context engineering、trade-off、spec、guardrails 等；首次出现不常见术语时，用中文解释其含义。不要强行翻译已经约定俗成的英文术语。
- 个人判断应具体、克制、基于原文证据。不要为了显得有观点而强行批判、拔高或制造冲突；如果原文主要价值是梳理清楚某个机制，也可以肯定其贡献并指出适用边界。

frontmatter 要求：

- `title` 优先采用“文章对象 / 核心主题 + 判断”的形式。标题应让读者知道这篇解读的核心观点，而不只是知道原文标题。
- `summary` 用 1-2 句说明这篇解读的核心价值，避免泛泛而谈。
- `description` 用一句话描述文章讨论范围，适合列表页和搜索摘要。
- `tags` 从既有 tag 体系中选 1-3 个。
- `date` 和 `lastmod` 使用当前时间，必须带时区。

链接格式：

- 正文开头第一次提到原文时，可以自然嵌入链接。
- 文章结尾必须保留 `## 原文`，并用列表链接到原文。
- 如果有多个来源，比如原文、作者补充 thread、官方文档等，结尾使用 `## 延伸阅读`。

禁止事项：

- 不要机械复刻原文目录。
- 不要编造原文没有的事实、数字、案例、产品行为或作者意图。
- 不要为了显得完整而堆砌背景知识。
- 不要用营销腔、夸张标题或空泛金句。
- 不要频繁使用 bullet list；列表只能服务于论点。
- 不要连续大段摘录原文，也不要只贴引用而不解释。
- 不要把历史 `sources/nlm/` 当作必须输入或唯一依据。

大纲规则：

- 普通文章可以直接写初稿。
- 原文特别长、主题重要、观点有争议、需要重写旧文，或用户明确要求深度解读时，先输出大纲并等待确认。
- 若原文特别长、机制复杂、观点分支多，writer 应先整理一份内部文章卡片，明确原文主张、关键机制、证据、trade-off 和拟采用的解读角度，再写正文。文章卡片不发布。

## 发布前检查 Prompt

请检查 `content/posts/<slug>.md` 是否可以发布。检查依据包括文章本身、`sources/orig/<slug>.md`，以及本文件的写作要求。

请按固定格式输出：

```markdown
## 结论

- PASS / NEEDS_REVISION

## 阻塞项

- 如果没有，写“无”。

## 建议项

- 如果没有，写“无”。

## 已直接修改

- 如果没有编辑，写“无”。
```

事实核对要求：

- 抽查文章中的关键事实、机制、数字、产品行为和作者观点，确认能在 `sources/orig/<slug>.md` 中找到依据。
- 如果是作者没有明说、但文章基于原文做出的推断，必须用“我的理解是 / 我认为 / 这意味着”等表达标清。
- 发现无法从原文支持的具体事实时，列为阻塞项。

阻塞项包括：

- 文章编造了原文不存在的事实、数字、案例、产品行为或作者观点。
- 关键判断无法从 `sources/orig/<slug>.md` 找到依据，且没有标明为个人推断。
- 文章写成了纯摘要、资料卡片或提纲式笔记。
- 缺少明确个人判断。
- 没有解释核心机制或关键论点。
- 缺少原文链接。
- frontmatter 缺失必要字段，或 `date` / `lastmod` 没有时区。

建议项包括：

- 开头判断不够直接。
- 小节标题太模板化。
- 列表过多，文章感不足。
- 背景铺垫过长。
- trade-off、限制条件或隐含前提还可以更具体。
- 个人判断可以更克制或更有依据。

## 可选 Subagent 流程

普通文章可以由当前 agent 直接写作并自查。

批量处理、长文、重要文章或改旧文时，建议拆成 writer + reviewer：

- Writer 必须阅读 `sources/orig/<slug>.md`。
- 默认只读 orig；只有改旧文或原文特别长时，才允许顺手读历史 `sources/nlm/<slug>.md`。
- 正式发布流程中，writer 应直接创建或更新 `content/posts/<slug>.md`，包含完整 frontmatter、正文和原文链接。
- 如果当前还处于讨论文章角度、标题或大纲阶段，writer 可以只输出草稿或大纲，不写文件。
- 普通文章可以直接写初稿。
- 原文特别长、机制复杂、观点分支多时，writer 应先整理内部文章卡片，再写正文。文章卡片不发布。
- Reviewer 按发布前检查清单验收，可直接做小修。
- Reviewer 可以直接修改错别字、重复句、过长句、明显弱的标题、缺少原文链接、frontmatter 小问题。
- 涉及核心判断、结构重组、事实依据不足的问题，reviewer 应退回 writer，而不是自行重写。
- Reviewer 发现建议项时，可直接小修或记录为建议，不阻塞发布。
- Reviewer 发现阻塞项时，如果是小问题可直接修；如果涉及核心判断、结构重组、事实依据不足，则退回 writer。
- Writer 修改后 reviewer 再检查。
- 同一篇文章最多进行两轮 writer-reviewer 循环；仍未 PASS 时，由主 agent 汇总争议点并做最终判断。
