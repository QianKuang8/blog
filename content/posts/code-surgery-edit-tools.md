Fabian Hertwig 这篇 [Code Surgery](https://fabianhertwig.com/blog/coding-assistants-file-edits/) 我觉得非常值得读，因为它不是在比较哪个 AI 编码工具更聪明，而是在讨论一个更底层的问题：为什么模型明明知道要改什么，最后却还是经常改不进去。

这篇文章真正抓住的核心，不是“代码生成”本身，而是**从模型理解到文件系统状态变化之间的交接问题**。只要这一层没有处理好，再强的模型也会在最后一步露怯，出现定位不到上下文、缩进错乱、补丁应用失败、甚至需要人工手动接管的情况。

## AI 编辑代码真正难的，不是理解，而是交接

很多人讨论 AI 编码时，天然会把注意力放在模型有没有理解需求、生成逻辑对不对。但这篇文章提醒我们，真正让产品变得可用的往往不是“想出来”，而是“落进去”。

模型没有直接文件系统访问权，它必须通过 patch、diff、search/replace、line operations 或 apply model 这类机制，把修改意图交给工具层，再由工具层去改变真实文件状态。问题就在于，这是一种间接状态操作，而任何间接操作都容易出错。

作者把这件事总结成 handoff，我觉得非常准确。因为这里的失败，并不总是推理失败，而往往是模型的内部表示没能准确映射到外部真实文件。它可能基于旧快照操作，也可能上下文锚点不够唯一，或者只是格式稍微不合规，最终就会导致整次编辑崩掉。

所以这篇文章最重要的价值，是把 AI 代码编辑重新定义成一个“状态交接系统”问题，而不是只把它看成文本生成问题。

## 五种编辑格式，其实代表五种工程哲学

文章把当前主流工具拆成了 Patch、Diff、Search/Replace、Line Operations 和 AI-Assisted Apply 五种路线。我觉得这里最值得看的，不只是术语分类，而是它们背后代表的不同工程哲学。

- **Patch / Diff**：尽量让模型输出一种结构化修改说明，靠上下文行去定位和应用
- **Search/Replace**：尽量把问题收缩成明确的 before/after 块
- **Line Operations**：用精确位置换取直接性，但脆弱性很高
- **AI-Assisted Apply**：承认普通规则系统不够稳，于是再引入一个专门的模型负责“应用”

把这几条路放在一起看，会发现它们其实都在回答同一个问题：如果模型不能直接写文件，那你到底要靠“更严格的格式”、还是“更灵活的匹配”、还是“再来一个专用模型”来把修改落进去？

这也是为什么我觉得这篇文章很适合作为总览。它不是告诉你哪条路绝对正确，而是把设计空间完整展开了。

## Codex、Aider、RooCode、OpenHands、Cursor，其实是在同一个坐标系里竞争

文章把几个工具并排看时，一个很明显的结论是：它们虽然表现形式不同，但竞争维度其实非常一致。

Codex 倾向于用上下文锚定 patch，通过渐进式匹配去提高应用稳定性；Aider 更像一个多格式实验场，既重视 diff 形式，也重视 flexible patching 的容错层；RooCode 用 fuzzy matching 和相对缩进映射，明显更偏“尽可能把修改救回来”；OpenHands 则在传统 patch 之外加入了 draft editor；Cursor 则进一步把 applying 独立成专门模型。

这让我觉得这篇文章最有启发的一点是：这些工具表面看像在做不同产品，实际上它们都在围绕同一条轴演化，也就是**从严格匹配到容错匹配，再到专门 apply 模型**。换句话说，行业并不是在各自乱走，而是在逐渐探索同一个瓶颈的不同解法。

## 分层匹配和错误反馈，是可用性的真正支柱

文章里有两条工程原则我非常认同：分层匹配，以及可操作反馈。

分层匹配的意义在于，系统不要一上来就过度宽松，也不要一失败就彻底报错。更好的路径是从严格上下文开始，逐步放宽到忽略空白、缩进映射、模糊相似度搜索。这样既保留了精度，又给系统留出自我修复空间。

而错误反馈的意义在于，一旦失败，不要只告诉模型“failure”。你必须告诉它到底哪里没对上，期待的上下文是什么，实际文件里出现了什么。只有这样，系统才有机会进入可迭代的恢复循环，而不是把失败原地丢回给人类。

我觉得这恰恰是很多产品体验差异的来源。用户感受到的是“这个工具老改不进去”，但工程上真正区分好的系统和差的系统的，往往正是这些恢复层设计得够不够细。

## 这篇文章最有价值的地方，是让“Apply”成为独立问题

我读这篇文章时一个非常强的感受是，它在不断把“应用修改”从一个次要细节提升成一个独立问题。无论是 Codex 的 patch engine、Aider 的 flexible patching、RooCode 的 fuzzy matching，还是 Cursor 的 apply model，都在说明同一件事：**apply 不是生成之后顺手完成的小动作，而是一块独立的工程领域。**

这件事非常重要，因为它和前面几篇关于 Aider、Cursor 的文章其实能连起来看。Aider 证明格式设计重要，Cursor 证明运行时和 apply 模型重要，而 Fabian 这篇文章则提供了一个更高层的视角：为什么这些系统最后都不得不在 apply 这一层投入巨大精力。

也就是说，AI 编码工具的下一阶段竞争，很可能越来越不只是“谁更会想”，而是“谁更会改进去，而且改进去之后还不弄坏周围状态”。

## 我的看法：AI 编码的核心竞争，正从生成质量转向交接可靠性

我读完之后最大的感受是，这篇文章帮我把很多零散观察统一了起来。以前大家总爱讨论模型写代码能力，但到了真正产品层，你会越来越发现，决定体验的经常不是“回答是否聪明”，而是“修改是否可靠地落盘”。

如果一个系统总是停在“我知道该怎么改，但没找到正确上下文”这一步，那它最终仍然只是建议引擎，而不是执行引擎。真正有价值的 AI coding tool，必须把这条 handoff 链路做扎实。

所以如果要总结这篇文章，我会说：它最重要的贡献，不是介绍了五种编辑格式，而是帮我们把问题重新看清楚了。AI 编辑代码真正难的，从来不只是生成，而是如何把模型里的意图，稳定、安全、可恢复地交接给真实文件系统。
