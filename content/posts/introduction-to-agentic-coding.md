Anthropic 这篇 [Introduction to agentic coding](https://claude.com/blog/introduction-to-agentic-coding) 我觉得很值得读，因为它并不是单纯在宣传 Claude Code，而是在试图给“AI 编程”这个领域重新划分阶段。过去几年我们讨论得最多的是补全好不好、聊天解释清不清楚，但这篇文章真正想推动的，是另一种理解：**编程辅助正在从局部建议工具，转向能够围绕整个代码库自主编排任务的系统。**

这件事的重要性在于，它改变的不是一个功能点，而是开发者与 AI 的合作关系。以前你负责组织任务，AI 负责局部产出；现在越来越多系统开始接手上下文搜集、计划调整、跨文件修改和验证闭环。这不只是“更强的补全”，而是整个编程工作流的边界在变。

## 真正的瓶颈，从来不是“不会写几行代码”

文章一开始讲得很清楚：AI 辅助编程每一代都在减轻某种认知负担，但早期方案也留下了新的税。自动补全解决的是“当前这一行怎么写更快”，对话式 AI 解决的是“一个问题能不能讲明白”，可一旦任务进入多文件协调、依赖传播、测试修复这类工程问题，真正折磨人的就不是代码生成本身，而是编排。

这也是我很认同这篇文章的地方。它没有把问题定义成“模型生成质量还不够高”，而是把矛头指向了更真实的痛点：开发者仍然要手动找上下文、手动拆任务、手动追依赖、手动把编译错误一轮轮喂回聊天框。也就是说，最大的开销不是写代码，而是**组织写代码这件事**。

从这个角度看，agentic coding 不是在替代编程，而是在替代编排税。

## Anthropic 真正想建立的是一条新的工作流

文章里把 agentic coding 拆成了四个阶段：上下文收集、自适应规划、实现循环、验证。我觉得这不是一个普通的流程图，而是在定义一种新的默认工作方式。

第一阶段要做的是把项目环境、代码结构、依赖关系和本地约定都读出来；第二阶段不只是列计划，而是根据环境反馈随时改计划；第三阶段开始真正跨文件动手；最后用测试和构建结果把系统重新拉回现实约束中。

如果把这四步连起来看，会发现它和传统的“我想一下、我写一点、我跑一下、我再修一下”当然很像，但差别在于，过去这些动作都由人手动串起来，而这里正在被打包成一个由 Agent 自主循环执行的系统。

我觉得这就是文章真正的新意：不是强调模型能生成更长的代码，而是强调它开始能接住整个开发闭环里原本最琐碎、最耗时、最反复的那部分工作。

## 终端优先，不是产品偏好，而是能力边界选择

这篇文章里我特别认同的一点，是它把 Claude Code 的终端优先设计讲得很明确。很多人会把 CLI 理解成一种品味选择，但从能力层面看，这其实是边界选择。

只要系统待在浏览器或受限 IDE 里，它能看到的世界始终是被截断的：文件视图是局部的，系统工具受限，构建和测试往往还得靠人手动接回去。终端则把 Agent 直接放进了项目真实运行环境里，让它能读文件、跑命令、执行测试、感知失败结果，再根据这些结果继续往下走。

这说明 Anthropic 对 agentic coding 的理解非常务实：真正有用的系统，必须能接触真实环境，而不只是理解一段被粘贴进来的代码。你让 AI 留在浅表交互层，它就只能做聪明助手；你把它放进终端和工具链，它才有机会进入真正的工程闭环。

## MCP 和 CLAUDE.md 的作用，是把 Agent 从“会写”变成“懂项目”

文章里还提到 MCP 和 `CLAUDE.md`，我觉得这两者非常关键，因为它们分别解决了 agentic coding 最容易缺失的两种上下文。

MCP 提供的是外部系统上下文，让 Agent 不只是看见本地文件，还能接 issue、文档、内部系统等更广的工作环境；`CLAUDE.md` 提供的是项目内约束，把命名习惯、代码标准、技术债和团队默认假设沉淀下来，避免每次会话都重新解释一遍。

这说明 Anthropic 并没有把 agentic coding 简化成“一个会调用 shell 的模型”，而是把它当成一个必须长期对齐团队环境的系统。真正能用的 coding agent，不只要会改代码，还要会在既有项目文化里改代码。

我觉得这点特别重要，因为很多人高估了“通用推理能力”，低估了“项目约束对实现质量的影响”。实际上，能不能模仿本地约定，往往比会不会写某段逻辑更决定最终产物是否可接受。

## Rakuten 案例真正证明的，不是模型神奇，而是闭环成立了

文章引用了 Rakuten 的案例：一个复杂、多语言、长时间运行的实现任务，最终由 Claude Code 在数小时内完成，且达到很高精度。很多人看到这种案例，第一反应是“模型太强了”。但我觉得更值得看的，是它说明整条 agentic loop 在真实工程任务里已经跑通了。

因为这种任务真正难的，不只是生成某个算法实现，而是要持续读环境、改代码、验证结果、根据反馈继续迭代。只要闭环里任何一环不成立，长时间 автономous work 就会很快崩掉。

所以我更愿意把这个案例看成对一种系统形态的验证，而不是单次模型表演。它证明的是：当上下文收集、规划、实现、验证这四步都能被打通时，AI 编程的单位就不再是“补一个函数”，而开始变成“把一个任务真正做完”。

## 我的看法：Agentic coding 真正改变的，是开发者如何分配注意力

我读完这篇文章最大的感受是，agentic coding 的真正变化不在于“AI 参与度更高”，而在于开发者的注意力配置开始改变。过去你得把很多脑力花在串流程、搬上下文、追依赖、跑测试上；现在这些劳动有机会被压到后台，让人类更专注于目标定义、约束设定和结果审查。

这不是说编程会被完全替代，而是说开发的劳动结构会被重新切分。对于复杂系统开发来说，最稀缺的人类能力不再只是写出某一段代码，而是判断什么值得做、什么算完成、什么符合项目方向。Agentic coding 正在把更多机械编排工作从人手上拿走，让这些判断层工作变得更突出。

所以如果要总结这篇文章，我会说：它真正有价值的地方，不是宣布“AI 开始自己写代码”，而是明确提出了一种新的编程协作模式。这个模式的核心不是更强的补全，而是**面向整个代码库的自主编排能力**。
