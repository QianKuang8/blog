Lance Martin 这篇 [Context Engineering for Agents](https://rlancemartin.github.io/2025/06/23/context_engineering/) 我觉得很值得读，因为它把很多原本分散的 Agent 技巧，重新串成了一张更清楚的系统地图。文章最有价值的地方，不只是列出几种做法，而是提出了一个很好用的类比：**把 LLM 看作 CPU，把上下文窗口看作 RAM。**

一旦接受这个类比，很多以前显得零散的经验就突然统一起来了。为什么长轨迹任务会越来越不稳定？为什么摘要、记忆、检索、子智能体这些机制会不断出现？因为本质上，你正在面对一个工作内存有限、但任务轨迹越来越长的系统，而真正的工程问题就是怎么管理这块内存。

## 长轨迹 Agent 的核心问题，其实是“内存污染”

文章一开始总结了四种很典型的失效现象：context poisoning、distraction、confusion、clash。我觉得这四个词总结得非常好，因为它们比“上下文太长了”更具体。

Poisoning 指的是幻觉或错误中间结果被继续写进后续轨迹，形成负反馈循环；distraction 是无关信息太多，让模型的注意力被稀释；confusion 是上下文里塞了太多相似但不必要的东西，比如重复工具说明、重叠指令；clash 则更严重，是不同上下文片段之间已经发生冲突，模型面对互相矛盾的信息不知道该信谁。

我觉得这套分类很有用，因为它提醒我们：长上下文的问题不只是“贵”或者“慢”，更关键的是它会改变模型的行为质量。上下文不是越多越安全，很多时候反而是越多越乱。

## Write、Select、Compress、Isolate，其实是一套完整的内存管理框架

文章把上下文工程拆成四类策略：Write、Select、Compress、Isolate。我很喜欢这个拆法，因为它不是按产品功能分，而是按系统如何处理信息流分。

- **Write** 是把信息写到上下文窗口之外，比如 scratchpad、memory、外部状态文件
- **Select** 是在运行时只取回当前步骤真正相关的部分
- **Compress** 是在保留关键信息的前提下对历史进行摘要和裁剪
- **Isolate** 是把不同任务、角色或状态拆到不同上下文单元里，避免互相污染

如果把这四个动作放在一起看，就会发现它们几乎覆盖了 Agent 系统所有关键设计选择。你是把历史继续塞进窗口，还是写到外部记忆？你是默认全量暴露工具，还是只按需选择？你是依赖统一大上下文，还是把复杂任务拆给多个子智能体？这些都可以放进这四个框架里理解。

## 这篇文章最好的地方，是把很多产品经验放到同一坐标系里

文章里提到了 Claude Code 的 auto-compact、ChatGPT 的长期记忆、Cursor 和 Windsurf 的代码检索、HuggingFace CodeAgent 的环境隔离。我觉得这很有价值，因为它说明这些看起来不相关的产品设计，其实都在回答同一个问题：**如何让有限的上下文窗口承载更长、更复杂的任务轨迹。**

比如 Claude Code 的自动压缩，解决的是历史持续膨胀；ChatGPT 的长期记忆，解决的是跨会话偏好保存；Cursor/Windsurf 的索引和重排序，解决的是代码库检索精度；CodeAgent 的环境隔离，解决的是让 token 密集对象不要直接占满主窗口。

这也是我认为这篇文章比普通“方法清单”更强的地方。它不是在堆案例，而是在给案例找共同的系统解释。

## 真正困难的不是“能不能压缩”，而是压缩以后还准不准

我觉得文章里一个很现实的提醒，是摘要和压缩并不是免费的。上下文压缩当然能省 token，但它经常会丢失长程依赖、细粒度约束和失败证据。尤其当任务本身需要跨很多轮次保持一致性时，摘要质量往往直接决定系统还能不能继续对。

这也是为什么文章会强调：多智能体虽然能隔离上下文，但 token 开销会明显增加；自动记忆虽然有用，但也会带来不可预测的注入风险。换句话说，context engineering 从来不是“选一个最强招式”，而是在不同约束下做取舍。

我很认同这种视角，因为它让上下文工程回到了工程本身，而不是又一次被包装成银弹。

## 我的看法：这篇文章把 Agent 问题重新翻译成了系统问题

我读完之后最大的感受是，它真正做成的一件事，是把很多原本容易玄学化的 Agent 讨论，重新翻译成了更接近系统工程的话语。

你不再只是讨论 prompt 怎么写、memory 要不要开、multi-agent 要不要上，而是开始讨论有限工作内存如何组织、哪些信息该持久化、哪些信息该延迟加载、哪些模块应该隔离。这个转换非常重要，因为它意味着 Agent 不再只是一个“会调用工具的大模型”，而越来越像一个需要认真做资源管理和状态管理的计算系统。

所以如果要总结这篇文章，我会说：它最有价值的地方，不是提供了一串术语，而是给 Agent 工程提供了一张可操作的内存管理地图。
