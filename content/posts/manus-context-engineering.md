## 核心论点：从微调到上下文工程

Manus 的架构演进反映了从端到端模型训练向高保真上下文工程的转变。催化因素：

- **反馈循环延迟**：历史模型微调需要数周迭代，而 prompt 和上下文操作可实现小时级迭代
- **模型过时**：为特定任务训练的定制模型被 GPT-3/Flan-T5 等前沿模型一夜淘汰
- **推理时适应性**：利用前沿模型的 ICL 能力，Agent 架构可与底层权重解耦

> 核心观点：将底层 LLM 视为与 Agent 架构**正交相关**。如果前沿模型进步是涨潮，Manus 是随潮水上升的船，而非固定在海底的柱子。

## KV-Cache 优化：生产 Agent 的生命线

生产 Agent 与标准聊天界面的关键差异：Agent 运行在迭代工具调用循环中，**输入/输出 token 比约 100:1**。预填充（prefill）阶段是主要计算瓶颈，维持稳定的 KV-Cache 是降低 TTFT 和成本的唯一可行路径。

### KV-Cache 优化清单

- **保持前缀稳定**：避免在 prompt 开头放置动态数据。自回归 LLM 中，序列开头一个 token 的偏移会**使后续整个缓存失效**。绝不在系统 prompt 中放高精度时间戳
- **只追加序列化**：禁止修改上下文历史中已有的 action/observation
- **确定性 JSON key 排序**：标准库不保证稳定排序，非确定性序列化会在不同分布式 worker 间静默破坏缓存
- **Session ID 路由**：使用 vLLM 等分布式推理框架时，确保同一 agent 循环的请求路由到持有相关缓存的同一 worker
- **策略性断点放置**：在系统 prompt 末尾手动插入缓存断点

### 缓存效果（Claude Sonnet 数据）

| 指标 | 未缓存 | 缓存 | 改善 |
|------|--------|------|------|
| 成本（每百万 token） | $3.00 | $0.30 | **10x 降本** |
| 延迟（TTFT） | 线性预填充 | 最小预填充开销 | 显著降低 |

## 工具管理：Logit Masking 而非动态加载

随着 Agent 能力扩展，出现"动作空间爆炸"。盲目提供数百个工具会增加次优工具选择和幻觉参数的概率。

### RAG 工具加载的失败

测试表明动态工具加载（通过 RAG 增删工具）适得其反。从当前上下文中移除已使用的工具会导致**模式违规或幻觉操作**——模型历史中仍包含对已不存在工具的引用。

### 上下文感知状态机方案

Manus 用**受限解码**替代上下文修改，通过 response pre-filling 操纵 logit processor，在不破坏 KV-Cache 的前提下将模型强制到特定模式：

- **AUTO 模式**：模型自主决定是否调用工具
- **REQUIRED 模式**：模型被强制调用工具，但自由选择工具
- **SPECIFIED 模式**：模型被限制在特定子集（如仅浏览器工具）

**Masking vs 移除的权衡**：Masking 保持 KV-Cache 稳定，防止历史工具引用混淆（50+ 步后仍稳定）；移除略微减少上下文长度但导致频繁缓存未命中和模式违规。

## 文件系统作为外部记忆

128K 上下文窗口不足以处理真实场景中的大型非结构化数据。性能退化（"Lost in the Middle"）和线性成本增长使超大上下文窗口效率低下。

### 记忆层级

1. **活跃上下文**：高速、高注意力成本（LLM Prompt）
2. **持久外部记忆**：容量无上限、结构化（沙盒文件系统）

### 可恢复压缩

将大型观测卸载到文件系统，在上下文中用**指针**（文件路径/URL）替代。这是"可恢复的"因为 Agent 可以学会按需将文件读回上下文。

> 理论视角：这一架构弥合了 Transformer Agent 与**状态空间模型 (SSM)** 的差距——通过将长期状态外部化到文件系统，Agent 克服了 SSM 缺乏全局注意力机制的限制，定位为**神经图灵机 (NTM)** 的功能继承者。

## 注意力操控：三大稳定模式

### 1. 目标复述（Goal Recitation）

维护 `todo.md` 文件。Agent 在每轮循环末尾重写当前计划并勾选完成项，将全局计划"复述"到模型最近的注意力范围内，防止长序列中的目标漂移。

### 2. 负面证据保留（Negative Evidence Retention）

显式**避免**清理错误或堆栈跟踪。保留失败尝试提供模型更新环境"信念"所需的负面证据，防止 Agent 重复相同失败操作进入无限循环。

### 3. 模式多样性（Pattern Diversity）

打破**少样本模仿陷阱**——Agent 因某种模式在历史中频繁出现而开始重复它。通过变换序列化模板和措辞引入结构化噪声，迫使模型对当前状态保持反应性。

## 架构决策矩阵

| 设计选择 | 主要收益 | 相关权衡 |
|---------|---------|---------|
| 上下文工程 > 微调 | 快速迭代；模型正交性 | 对 prompt 结构和前沿模型推理的高度敏感 |
| Logit Masking / Pre-filling | 稳定 KV-Cache；10x 降本 | 工具需要严格命名约定（如 `browser_` 前缀） |
| 文件系统即记忆 | 突破上下文窗口限制 | Agent 需学会显式读写记忆管理 |
| 负面证据保留 | 错误恢复；防止重复失败循环 | 增加上下文长度和累计 token 消耗 |
| 目标复述（todo.md） | 防止 50+ 步循环中的目标漂移 | 每次循环迭代增加辅助管理步骤 |
