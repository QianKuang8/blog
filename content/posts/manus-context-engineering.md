Manus 这篇 [AI 智能体的上下文工程：构建 Manus 的经验教训](https://manus.im/zh-cn/blog/Context-Engineering-for-AI-Agents-Lessons-from-Building-Manus) 我觉得很有价值，因为它把 Agent 讨论从“模型选哪个、prompt 怎么写”继续往下推进了一层，直接落到了运行时工程。文章真正想说的其实不是一句泛泛的“上下文工程很重要”，而是：**当 Agent 开始长时间、多轮次、重工具调用地运行时，上下文本身就会变成系统最核心的基础设施。**

这篇文章之所以重要，是因为它把很多以前看起来像“调参技巧”的东西，重新解释成了工程主线。KV-Cache、append-only 历史、文件系统作为记忆、错误保留、目标复述，这些都不是小优化，而是在决定一个 Agent 是否真的能稳定跑下去。

## Manus 真正放弃的，不是微调，而是慢反馈循环

文章一开头先讲为什么 Manus 押注上下文工程而不是端到端训练，我觉得这个判断非常现实。问题并不只是“前沿模型更强”，而是微调路线的反馈回路在 PMF 之前太慢了。你改一点逻辑，要等数周才能重新训练和评估，这对快速迭代的产品来说几乎是致命的。

而上下文工程的吸引力，就在于它能把改动周期压到小时级。更重要的是，它让产品和底层模型解耦：模型换代了，你不用推倒整个系统，只要你的架构足够围绕上下文组织，而不是围绕特定权重绑死。

我很认同作者那句类比：如果模型进步是涨潮，产品最好是一条船，而不是固定在海床上的柱子。因为到了 2025 这个阶段，模型更新速度本身已经成为架构设计约束的一部分。

## 文章真正的核心指标，其实是 KV-Cache 命中率

如果要从这篇文章里只记住一个技术点，我会选 KV-Cache hit rate。作者甚至直接说，这是生产 Agent 最重要的单一指标。我觉得这个判断非常值得认真对待。

原因很简单：Agent 和普通聊天不一样。它运行在 action-observation 循环里，输入会越滚越长，而输出往往只是简短工具调用。于是系统的主要成本和延迟压力并不在解码，而在 prefill。只要缓存命中率差，TTFT 和成本都会迅速恶化。

这就把很多本来容易被当成细节的事重新抬到了战略高度：

- 提示前缀要稳定
- 历史必须 append-only
- JSON 序列化要确定性
- 分布式推理时要固定会话路由

我特别喜欢这部分，因为它让人意识到，Agent 的“聪明”并不只建立在更好的推理上，还建立在系统有没有能力避免重复付出同一份上下文代价。

## 为什么工具不能随便动态增删

文章里另一个很有意思的点，是 Manus 对动态工具加载的反思。直觉上看，工具太多了，当然应该通过 RAG 或其它方法按需加载；但 Manus 的经验恰恰说明，这样做经常适得其反。

理由其实很硬：工具定义通常在上下文前部，一旦改动，就会让缓存失效；更糟的是，历史里已经出现过的工具调用还留在那里，如果当前上下文里这些工具突然消失，模型就会在记忆和现实之间产生冲突，最终引发 schema 违规或幻觉动作。

所以 Manus 选择的不是移除工具，而是用 logit masking 和 response prefilling 去限制模型当前可选动作。这个选择我觉得很有代表性，因为它说明在 Agent 系统里，“理论上更干净的上下文”未必比“运行时更稳定的上下文”更有价值。

换句话说，系统设计开始从“最优表达”转向“最优稳定性”。这非常像一个成熟 runtime 团队会做的取舍。

## 文件系统即记忆，意味着上下文窗口不再是唯一记忆容器

我觉得这篇文章最有启发的一点，是它把文件系统明确提升成了 Agent 的外部记忆层。很多人看到 128K、200K 甚至更长的上下文窗口，会本能地觉得“够用了”，但 Manus 这里很清楚地指出：真实 Agent 面对网页、PDF、长链任务时，再大的窗口也很快会变得昂贵而脆弱。

所以他们采取的路线不是一味堆上下文，而是把大块观测卸载到文件系统，在上下文里保留可恢复指针。这个做法的妙处在于，它不只是压缩，而是**可恢复压缩**：信息不在当前 prompt 里，但并没有消失，Agent 随时可以按需读回来。

我觉得这是 Agent 迈向真正长期运行的关键一步。因为一旦你接受“活跃上下文”和“持久记忆”应该分层，很多设计都会自然变得清晰：上下文负责当前高注意力状态，文件系统负责无上限持久存储。

从这个意义上说，Manus 这篇文章并不是在教人“怎么写更长 prompt”，而是在教人如何设计一个外部记忆架构。

## 注意力操控并不玄学，它是长链 Agent 的行为稳定器

文章后面几节讲 goal recitation、negative evidence retention 和 pattern diversity，我觉得这三点放在一起看特别好。它们共同说明了一件事：长链 Agent 不只是要“有上下文”，还要被系统性地引导注意力。

目标复述通过 `todo.md` 反复把当前任务拉回上下文末尾，防止系统在 50 步以后忘记自己本来要干什么；保留错误则让模型真正看到失败证据，而不是把失败痕迹清掉后假装世界没变；模式多样性则是在防止模型被自己历史中的单一格式和行为习惯困住。

这几件事单独看像技巧，合起来看其实是在构建一种行为稳定器。因为长序列里最容易发生的，不是模型突然变笨，而是它开始机械重复、目标漂移、或者忽略负面证据继续撞墙。

我觉得 Manus 的经验很重要，因为它说明：Agent 的行为控制不一定靠更复杂模型，也可以靠更精细的上下文编排来完成。

## 我的看法：Agent 的竞争，正在从 prompt 技巧转向运行时工程

我读完这篇文章后最大的感受是，Agent 系统的真正竞争焦点正在发生转移。以前大家很爱讨论 prompt 写法、tool design、workflow 模式，这些当然还重要，但到了长时间运行、多轮工具调用的阶段，真正拉开差距的开始变成运行时工程。

谁能把缓存打稳，谁能把工具空间控制住，谁能把外部记忆和活跃上下文分层，谁能让模型在几十轮之后仍然记得目标并从失败里学会调整，这些东西会越来越决定一个 Agent 是否可用。

所以如果要总结 Manus 这篇文章，我会说：它最有价值的地方，不是告诉你“上下文工程很重要”，而是把这件事拆成了一套非常具体的运行时原则。到了这个阶段，Agent 已经不只是 prompt 产品，而越来越像一个围绕上下文、缓存和记忆精细设计出来的系统。
