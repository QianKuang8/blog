LangChain 这篇 [The rise of "context engineering"](https://blog.langchain.com/the-rise-of-context-engineering/) 我觉得很值得读，因为它做的并不是再发明一个新术语，而是在给 2026 年的 Agent 工程现实补一套更准确的语言。文章真正想纠正的一件事是：**很多 Agent 的失败，并不是因为模型不会，而是因为系统没有把它完成任务所需的上下文组织好。**

这点听起来像一句常识，但我觉得它的重要性在于，它改变了我们排查问题的默认方向。以前系统效果不好，大家第一反应常常是换模型、调 prompt、加 few-shot；而这篇文章的视角是，先别急着怪模型，先问一句：你到底有没有把它需要的信息、工具和格式，在正确时机交给它？

## 这篇文章真正改写的，是失败归因

文章里我最认同的一点，是它把“模型能力不足”和“上下文构造失误”明确区分开了。这个区分很关键，因为两类问题的修法完全不一样。

如果模型已经拿到了完成任务所需的所有信息，但依然做不好，那才是模型能力边界；但现实里更常见的情况，其实是系统在运行时根本没有把关键背景、当前状态、工具入口或者约束条件传进去。模型不是读心者，它只能在你实际给到的输入上工作。

所以这篇文章真正有价值的地方，不是说“上下文很重要”，而是重新划定了工程责任边界：**只要系统还没把上下文组织到位，就不能轻易把失败归咎给模型。**

## Context engineering 不是 prompt engineering 的换皮

很多人第一次看到这个词，会下意识把它理解成“更复杂的 prompt engineering”。但文章其实在推动一个更大的转向：从静态提示词优化，转向动态运行时编排。

Prompt engineering 更像是在调一条输入字符串，关注的是措辞、顺序、示例和输出格式；而 context engineering 关注的是系统如何在执行过程中动态决定：

- 现在该给模型哪些信息
- 哪些工具应该暴露给它
- 工具返回结果要用什么格式表达
- 哪些历史上下文应该保留，哪些应该裁掉

也就是说，这已经不是“写 prompt”的问题，而是“构建一个上下文供给系统”的问题。到了 Agent 阶段，真正重要的并不是某一句魔法咒语，而是整个系统有没有能力持续给模型喂正确的工作材料。

## 正确的信息、工具和格式，缺一不可

文章把 context engineering 拆得很清楚：right information、right tools、right format。我觉得这个拆法非常实用，因为它直接对应三类最常见的生产故障。

第一类是没有给到足够的信息。模型看似胡说，实际上只是你没把必要背景传进去。第二类是没有给到正确的工具。任务本来就需要查数据、执行操作或读外部状态，你却希望模型只靠上下文窗口硬推出来。第三类则是格式问题，信息虽然给了，但以一种极其不利于模型消费的方式给了，比如大段原始 JSON、混乱日志、没有重点的错误输出。

我很认同这里的判断，因为它说明 Agent 工程开始更像接口设计，而不是写文案。你不是在“说服”模型，而是在给它搭一个能够稳定理解和行动的工作界面。

## 这篇文章背后，其实是在强调“拥有自己的上下文系统”

文章虽然没有极端地反框架，但它隐含的价值观我觉得很清楚：上下文构建逻辑不能完全委托给黑盒。

因为一旦你开始认真做 Agent，就会发现真正决定质量的往往不是框架 API 漂不漂亮，而是你能不能控制：

- 进入模型的到底是什么
- 为什么是这些而不是那些
- 失败时能不能追溯是哪一段上下文出了问题

这也是为什么像 LangGraph、LangSmith 这类东西在这套语境里变得更重要。它们真正提供的不是“更高级的 Agent 抽象”，而是更强的运行时可见性和控制权。没有这层控制，所谓 context engineering 最后很容易退化成另一种玄学调参。

## 我的看法：这篇文章的重要性，在于它帮 Agent 工程换了一个主语

我读完之后最大的感受是，这篇文章并不是在讲一个新技巧，而是在帮整个行业把主语从“prompt”换成“system”。

过去几年很多讨论都默认：只要模型够强，再加一点 prompt 技巧，Agent 就会自然变好。但真正做过长轨迹、多工具、重状态系统的人都知道，瓶颈往往不在生成瞬间，而在运行时的上下文组织能力。谁能更稳定地把正确的信息、正确的工具和正确的格式在正确时机交给模型，谁就更可能把 Agent 做稳。

所以如果要用一句话总结这篇文章，我会说：它真正提出的，不是一个新流行词，而是一种新的工程判断。**Agent 的竞争，正在越来越多地落到上下文系统本身，而不只是模型参数和 prompt 技巧。**
