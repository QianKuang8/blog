周星星这篇 [Agent 元年复盘：从 Claude Code 到 Deep Agent，Agent 的架构之争已经结束](https://zhuanlan.zhihu.com/p/1983512173549483912) 最值得读的地方，不是它继续重复“2025 是 Agent 元年”这种大叙事，而是它给了一个更具体的判断：到了这个阶段，行业真正有价值的问题已经不再是“Agent 要不要做”，而是“它最终收敛成什么形态，以及还剩下哪些难题没有解决”。

如果把这篇文章的主线压缩一下，我会把它概括成两句话：第一，通用型 Agent 的基础架构已经明显收敛到主 Agent 调度、子 Agent 执行的形态；第二，架构不再是最主要的瓶颈，真正难的地方开始转向业务知识如何注入，以及 long-running 系统怎么稳定跑下去。

## 这篇文章真正想下的结论，是“架构之争已定”

文章的判断非常明确：以 Claude Code 和 Deep Agent 为代表的通用型 Agent，已经让行业的主流形态逐渐收敛。这里说的“收敛”，并不是大家长得一模一样，而是说一套核心骨架已经越来越清晰：

- 一个强主 Agent 负责理解任务、规划步骤和调度资源
- 必要时生成子 Agent 处理局部任务
- 文件系统成为共享工作区和上下文卸载层
- 系统提示承担越来越多复杂行为约束

这个判断我基本认同。因为 2025 年很多架构讨论，看起来像是在比较不同框架，实际上真正保留下来的共识都越来越像：单体全能 Agent 不好维护，上下文容易污染，任务一长就会发散；而主从式结构至少能把规划、执行、记忆和上下文隔离这几件事安排得更像一个系统。

所以这篇文章最重要的价值，不是展示一个新架构，而是把“什么会留下来”说清楚了。

## 它背后真正的哲学转向，是从消除方差变成管理风险

文中那句 “We are managing risk, not eliminating variance.” 我觉得是全篇最关键的底层判断。因为它点穿了 Workflow 和 Agent 之间真正的差别。

Workflow 的思路是：把业务逻辑拆成显式流程，尽量把方差消灭在执行之前。Agent 的思路则是：承认真实世界的边缘情况根本消灭不完，于是用自然语言抽象、上下文工程和 test-time token 消耗去管理这种风险。

这也是为什么作者强调，复杂度并没有消失，只是从图结构和流程编排转移到了 prompt、skills、上下文管理和系统提示里。这个说法我很认同。很多人误以为 Agent 是“少写流程、更多智能”，其实更准确的说法是：**流程复杂度被搬了家**。

从工程视角看，这个转向意味着你不再追求绝对确定性，而是追求一个更高上限但可被约束的系统。这件事的代价很高，但也正是 Agent 真正有吸引力的地方。

## Deep Agent 的门槛，不在“会不会调用工具”，而在“够不够垂、能不能长跑”

作者把 Deep Agent 的要求归结成两个维度：够垂和 long-running。我觉得这个拆法很有效，因为它一下就把“通用演示能力”和“生产可用能力”分开了。

所谓够垂，本质上不是多接了几个行业 API，而是 Agent 是否真正吃进了某个行业的判断标准、潜规则和成功经验。招聘 Agent 要像资深招聘经理，营销 Agent 要能给出与市场团队高度重合的 KOL 筛选和报价，这些都不是“多搜一点资料”就能做到的，而是业务知识真的被结构化进系统之后才可能达成。

而 long-running 维度则说明另一件事：Agent 不能只会完成一个短任务 demo，它还必须能在长时间、多步骤、多工具调用的条件下保持稳定和方向感。也就是说，真正能用的 Agent，不只是会想，还得能持续做。

把这两个条件放在一起看，Deep Agent 其实是在抬高门槛：不是任何能跑几个 tool call 的系统都能叫 Deep Agent。它要求的是“专业判断 + 长链执行”同时成立。

## Skills、文件系统和主从结构，其实在共同解决同一件事

这篇文章花了很多篇幅讲 Skills、Sub-Agents、File System 和复杂系统提示。表面上看是几个分散的话题，但我觉得它们其实都在共同解决一个问题：**如何在不把主上下文塞爆的情况下，让 Agent 既有足够的知识，又能跑得足够久**。

Skills 负责按需加载业务知识，而不是一开始把所有东西暴力塞进 prompt；子 Agent 负责上下文隔离，把局部任务从主循环里切出去；文件系统负责把中间状态、笔记和工作产物卸载到外部介质；系统提示则负责规定什么时候规划、什么时候委派、什么时候自己做。

从这个角度看，整个主从架构的真正目的，不只是“并行”或者“更强”，而是要对上下文窗口做精细化治理。因为只要任务一变长，真正首先崩掉的通常不是模型能力本身，而是上下文组织方式。

这也是为什么我觉得这篇文章讲 Skills 很有意思。它本质上是在把业务知识封装成分层可发现的文件系统，而不是继续押注“更大的 RAG 一锅炖”。这个思路非常符合 2025 年很多 Agent 产品真正走通的方向。

## 工具分层和自动压缩，说明 Agent 已经进入运行时优化阶段

文中还有两个我很认同的点：工具分层设计，以及自动压缩。

工具分层的逻辑很简单但很实用。不是为每个能力都单独注册一个 tool，而是把高频核心工具保持在一个有限集合里，把低频复杂能力下沉到 shell 或脚本层，通过代码封装去减少 LLM 回合数。这种设计不是为了优雅，而是为了降低上下文混淆和调用成本。

自动压缩也是同样的思路。只要任务足够长，token 迟早会满；而一旦上下文逼近上限，系统就必须主动总结前文，把重要状态压缩下来，不然 long-running 只是口号。也就是说，到这个阶段，Agent 已经不再只是“推理架构”问题，而开始明显进入运行时优化问题。

这让我觉得本文最有现实感的一点在于，它没有把 Deep Agent 讲成某种神奇智能，而是很诚实地把问题落在上下文、缓存、工具负载和运行稳定性上。真正的系统就是在这些地方拉开差距的。

## 我的看法：通用型 Agent 的分水岭，正在从架构设计转向业务沉淀

我读完之后最大的感受是，2025 年 Agent 行业的一个明显变化，就是“架构感”开始弱下来，“业务感”开始强起来。架构之争当然还存在，但在主从模式、上下文压缩、文件系统、技能化知识这些方向上，主流共识其实已经很接近了。

真正开始拉开差距的，是谁能把行业知识更丝滑地沉进 Agent，谁能把一个系统从“会跑”变成“可信、可持续、可长跑”。换句话说，未来竞争的重点未必是再造一个更复杂的框架，而是把具体业务的 SOP、判断标准和工具链，转化成 Agent 真的能稳定使用的工作流资产。

所以如果要总结这篇文章，我会说它最重要的地方不是告诉你“Deep Agent 多强”，而是告诉你：当基础架构逐渐收敛之后，真正的挑战会转移到知识沉淀、运行稳定性和业务改造能力上。到了这个阶段，Agent 的胜负已经不只属于模型团队，而属于那些能把业务真正重构为 Agent 友好系统的团队。
