OpenAI 这篇 [Harness engineering: leveraging Codex in an agent-first world](https://openai.com/zh-Hans-CN/index/harness-engineering/) 很容易被读成一篇“Codex 写了百万行代码”的故事，但我觉得它真正重要的地方不在这个数字，而在于它展示了 agent-first 软件工程会把组织的注意力转移到哪里。

文章描述的实验很激进：一个内部 beta 产品，从空 Git 仓库开始，五个月后达到约一百万行代码；应用逻辑、测试、CI、文档、可观测性和内部工具都由 Codex 编写；三名工程师推动约 1500 个 PR，后来团队扩到七人后吞吐量还继续上升。核心约束是“不手动写代码”。

这个约束不是噱头，而是为了逼出一个问题：如果人类不再把主要时间花在敲代码上，那么工程系统必须长成什么样，agent 才能可靠地产出真实产品？

## 工程师的工作从写代码变成设计条件

文章反复强调一句话：人类掌舵，智能体执行。这里的“掌舵”不是简单写需求让 Codex 生成代码，而是设计环境、明确意图、搭建反馈回路，让 agent 能够自己推进工作。

早期进展并不快。原因不是 Codex 没能力，而是环境不够明确：缺工具、缺抽象、缺约束、缺可读结构。于是人的工作从“我来实现这个功能”变成了“agent 还缺什么能力，怎样把这个能力做成可发现、可执行、可验证的系统条件”。

这个角色变化很关键。传统工程师经常在问题上直接出手：改代码、修测试、补脚本。agent-first 的做法则把直接出手视为失败信号，因为它没有改善系统下一次完成类似任务的能力。更高杠杆的做法，是把缺失的工具、文档、lint、测试、技能或流程补进仓库，让 Codex 以后能自己用。

## 让应用对 agent 可读

随着代码吞吐量上升，OpenAI 团队遇到的瓶颈变成了人工 QA。解决方向不是让人类看更多页面，而是让应用本身对 Codex 可读。

文章给了两个具体例子。第一，应用可以根据 git worktree 启动，Codex 能为每个变更运行独立实例。团队把 Chrome DevTools Protocol 接入 agent runtime，并提供 DOM 快照、截图和导航技能，让 Codex 可以复现 UI bug、验证修复、推理界面行为。

第二，可观测性也被本地化并暴露给 Codex。每个 worktree 都可以有临时日志、指标和 trace，任务结束后删除。Codex 能用 LogQL 查询日志、用 PromQL 查询指标，于是像“服务启动必须在 800ms 内完成”或“关键用户旅程 span 不超过两秒”这样的目标，才有可能被 agent 自己验证。

我觉得这是文章最实用的部分。很多团队把 agent 接入代码仓库后，仍然让它停留在文本和文件层；但真实软件质量常常体现在运行时。只有把 UI、日志、指标、trace、测试和部署反馈变成 agent 可读的接口，agent 才能从“生成代码”升级为“验证行为”。

## 仓库是记录系统，不是上下文垃圾桶

文章对知识管理的反思也很有价值。OpenAI 团队尝试过一个巨大的 `AGENTS.md`，结果失败了：上下文太长会挤掉任务和代码，过多指导会互相稀释，文档会腐烂，也很难机械检查。

他们最后把 `AGENTS.md` 缩成目录，而不是百科全书。真正的知识放在结构化 `docs/` 里：设计文档、执行计划、产品规格、架构文档、质量评分、可靠性和安全文档等。agent 从一个小入口开始，通过渐进式披露找到更深的信息。

这和人类工程组织很像。新人不需要一开始读完所有公司知识，而需要一张可靠地图，知道遇到某类问题时去哪看。对 Codex 来说也一样：运行时上下文里无法访问的信息，就等于不存在。Slack 讨论、Google Docs、人的脑内经验，如果没有进入版本化仓库，就不能被 agent 稳定利用。

更进一步，OpenAI 还用 linter 和 CI 检查知识库是否更新、交叉链接是否正确、结构是否符合要求，并用周期性的 doc-gardening agent 修过时文档。这说明 agent-first 的文档不是“写给人看的补充材料”，而是运行系统的一部分。

## 约束比微观管理更重要

完全由 agent 生成代码，最大风险之一是架构漂移。文章给出的解决方式不是不断在 prompt 里要求“写得更好”，而是把不变量编码进系统。

他们围绕固定的业务域分层建立架构约束，例如 Types、Config、Repo、Service、Runtime、UI 这样的依赖方向，并通过自定义 linter 和结构测试机械执行。结构化日志、模式和类型命名、文件大小、平台可靠性要求，也通过规则固化下来。甚至 lint 错误信息本身都会面向 agent 编写修复指令。

这个思路很重要：agent 的速度越快，人工 review 越跟不上，越需要把关键品味和边界转化成可执行规则。对人类团队来说，一些规则可能显得繁琐；但对 agent 系统来说，它们是速度的前提。因为规则一旦编码，就能在所有未来运行中重复生效。

同时，文章并不主张到处限制。它强调的是在中心层强制边界，在局部层允许自由。只要输出正确、可维护、对未来 agent 清晰可读，就不必要求生成代码完全符合某个人类的表达偏好。

## 吞吐量改变合并和债务治理

当 Codex 的吞吐量远高于人类注意力时，传统合并哲学也会变化。文章提到他们会尽量减少阻塞合并门，PR 生命周期很短，偶发测试失败通常通过后续重跑解决，而不是长期卡住进展。

这不是所有团队都能照搬的策略。它成立的前提是纠错成本低、反馈回路快、自动化验证和后续清理足够强。如果缺少这些支撑，低门槛合并只会制造混乱。

文章里的另一套机制是“垃圾收集”。完全自主的 agent 会复现仓库已有模式，包括不理想的模式，长期会产生漂移。团队一开始每周花一天人工清理，后来改成把“黄金原则”编码到仓库中，并定期运行后台 Codex 任务扫描偏差、更新质量评级、发起定向重构 PR。

我觉得这个比“一百万行代码”更值得关注。agent-first 系统不是没有技术债，而是必须把债务治理变成持续的小循环。否则 agent 只会更快地放大坏模式。

## 我的判断：harness engineering 是组织工程

这篇文章把 harness engineering 从“模型外壳”扩展成了组织工程。它包括运行环境、仓库知识、工具接口、验证系统、架构约束、文档维护、PR 流程和债务回收。模型当然重要，但真正决定能否放大人类时间的，是这些支撑结构。

文章也给出了明确边界：这种端到端自主能力高度依赖特定仓库结构和工具投入，不应在没有类似投资时假设可以泛化。这一点很克制，也很关键。很多团队如果只复制“不手写代码”的口号，而没有复制可读性、可观测性、约束和反馈回路，结果只会是更快地产生不可控复杂度。

我的理解是，agent-first 软件工程不会取消工程纪律，只会改变纪律出现的位置。过去纪律体现在人如何写代码、review 代码；现在越来越多地体现在环境如何限制错误、如何提供上下文、如何让 agent 自证、如何把人的判断固化成可重复机制。

这也是 harness engineering 真正值得讨论的原因：它不是让 agent 更像人，而是让软件组织学会围绕 agent 重新设计工作系统。
