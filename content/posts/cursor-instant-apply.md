Cursor 这篇 [Near-Instant Full-File Edits](https://web.archive.org/web/20240823050616/https://www.cursor.com/blog/instant-apply) 我觉得最值得看的一点，不是它宣布自己把编辑速度做到了 1000 tokens/s，而是它非常明确地告诉你：代码编辑其实不是一个单阶段问题。真正的系统应该把“知道要改什么”和“把改动机械地、完整地落到文件里”分开处理。

这件事很关键。因为很多前沿模型在对话里看起来很聪明，但一旦进入大规模文件编辑，就会暴露出完全不同的一面：会省略代码、会顺手改掉无关内容、会因为延迟太高把整个使用体验拖垮。Cursor 这篇文章的贡献，是把这些问题重新归类成运行时问题和应用层问题，而不只是笼统归为“模型还不够好”。

## 真正被修复的，是 frontier model 的“编辑差距”

文章把问题讲得很直接：前沿模型很会推理，但不代表它们很会机械地完成大文件编辑。你让它规划，它可能很好；你让它把几百行代码无错地完整落盘，它就开始暴露出另一种短板。

这类短板主要体现在三件事上：

- 懒，喜欢省略和留占位符
- 不稳，会顺手做无关改动或引入语法问题
- 慢，大文件编辑打断心流

这让我觉得 Cursor 真正抓到的问题不是“代码生成效果差”，而是“编辑执行层和推理层并不是同一种能力”。很多团队会天然把这两个阶段混在一起，但 Cursor 很早就把它们拆开了。

## Plan and Apply 的拆分，其实是在重新分配模型职责

Cursor 的两阶段架构非常值得记住。第一阶段由高能力 frontier model 负责规划，也就是跟用户对齐意图、理解修改目标；第二阶段则交给专门的 fast-apply 模型，去把这些修改真正写回当前文件。

我觉得这套拆法的价值，不只是“更快”，而是它承认了不同模型在不同环节里的最优角色并不一样。会聊天、会规划、会抽象高层逻辑的模型，并不一定适合做机械应用；而执行层真正要的，往往是稳定、完整、少犯低级错误。

从这个角度看，Cursor 并不是简单做了一个更快的 apply，而是在做职责分离：让 frontier model 保持它在 reasoning 层的优势，让 specialized model 去接住最容易把体验搞坏的那一段。

## 全文件重写听起来笨，但它可能更符合模型分布

文章里最有意思的争议点，是 Cursor 明明知道 diff 更省 token，却依然把大量精力押在全文件重写上。我觉得这个选择特别有代表性，因为它体现了一个常常违背直觉的事实：**token 更省，并不一定意味着系统更强**。

Cursor 的逻辑是，模型在训练时见过更多完整代码文件，而不是复杂 diff；同时，全文件重写虽然输出更长，但也给了模型更多 forward passes，相当于给了它更多“思考步数”。如果你把任务压缩成一个过于紧凑的 diff，模型反而更容易在格式、定位或细节上出错。

我觉得这个判断很值得重视。它说明编辑系统的优化目标不应该只盯着 token 效率，而应该同时看分布对齐、错误率和最终交付质量。很多时候，“更像模型熟悉的文本分布”比“更省 token”更重要。

## Speculative Edits 的真正意义，是把“已有文件”变成强先验

Cursor 这篇文章最亮眼的技术点当然是 speculative edits，但我觉得它真正的价值不在于速度数字本身，而在于它利用了代码编辑这个任务特有的性质：原文件里已经存在大量正确 token，它们本身就是最好的草稿。

传统 speculative decoding 要依赖一个 draft model；Cursor 的思路则是，代码编辑和自由生成不一样，当前文件状态已经构成了强先验，所以可以用确定性算法直接从现有内容里推测未来 token，而不是再引入一个额外模型。

这件事的意义非常大，因为它把编辑任务的本体优势用到了运行时里。代码编辑不是从零开始写，而是在一个高重合度文本上做局部变形。只要系统真的利用好这点，速度就会出现数量级变化。

所以这项工作给我的感觉是：它不是单纯在优化解码，而是在把“编辑任务和自由生成不同”这个事实，彻底工程化。

## 这篇文章也在反过来证明：diff 路线并不是默认赢家

文章里还有一个我很认同的地方，就是它没有默认认为 diff 就一定优于 full rewrite。很多人看到 token 开销会本能得出结论：当然应该让模型输出 patch。但 Cursor 的实验恰恰在提示我们，真正的问题不是“哪种格式理论上更紧凑”，而是“哪种格式在当前模型分布和运行时条件下更可靠”。

这也和 Aider 那篇 unified diff 形成了一个很有意思的对照。Aider 证明了好的 diff 格式能显著改善模型表现；Cursor 则说明，在另一套模型和系统架构里，全文件重写加专门 apply 模型和 speculative edits 也完全可能成为更优路线。

我觉得这两个结论并不冲突。它们共同说明的是：编辑系统的核心竞争，不在“diff vs rewrite”谁有天然真理，而在于你有没有围绕自己的模型、延迟预算和应用层能力，把整条链路做成闭环。

## 我的看法：Cursor 真正展示的是“运行时工程”开始主导体验

我读完之后最大的感受是，Cursor 这篇文章其实没有在单纯讨论模型，而是在非常明确地强调运行时工程的地位。数据怎么构造、模型怎么分工、推理如何加速、为什么要把 planning 和 applying 拆开，这些东西共同决定了最终体验是不是“instant”。

这和很多人理解中的 AI 编码产品很不一样。很多人会觉得核心竞争还是谁家模型更强，但从这篇文章看，至少在编辑层，真正拉开体验差距的已经越来越像是系统工程能力：谁能把 apply 做得更稳、更快、更少打断人。

所以如果要总结这篇文章，我会说：Cursor 最重要的贡献，不是证明“我们很快”，而是证明了一个方向：**在代码编辑这件事上，规划与应用应该被拆开，全文件重写也可以在足够强的运行时优化下，变成一条非常有竞争力的工程路线。**
