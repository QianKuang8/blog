这篇 [年终总结：2025年 AI 产品及架构演进深度研究报告](https://blog.zihanjian.com/article/2b69db82-fa86-80f4-abc2-f6a70258f7ad) 很长，也很杂，但它最有价值的地方恰恰在于把 2025 年很多看似分散的现象串成了一条线：为什么企业对 Copilot 的热情开始降温，却又在下半年迅速重新拥抱 Agent；为什么单体 ReAct 架构越来越显得吃力；为什么多 Agent、事件驱动和时序记忆开始同时变热。

如果只看表面，这像是一篇趋势综述；但如果抓住主线，它其实在讲一个更明确的变化：**AI 产品终于从“会生成内容的助手”开始转向“能持续执行任务的系统”**。而一旦目标从回答问题变成交付结果，产品形态、工程架构、记忆机制和商业模式都会一起变。

## 2025 年最重要的变化，是 Copilot 模式开始失效

文章一开始提到所谓 “Pilot Purgatory”，我觉得这是 2025 年很多企业 AI 项目最准确的写照。2024 年大家大量上线 Copilot，热闹也确实热闹，但真正落到业务主流程里的并不多。多数系统能做的还是文案草拟、会议总结、资料整理这类外围辅助工作，离真正“替你把事情做完”差得很远。

问题的根子并不神秘。Copilot 模式本质上还是一种高认知负荷交互：人要持续写 prompt、补上下文、盯执行过程、自己判断什么时候结束。系统看起来很聪明，但真正承担流程责任的还是人。只要业务链条一复杂，这种模式就会迅速暴露出边界。

所以 2025 年真正发生的不是 AI 热度消失，而是企业开始更清楚地区分两件事：一个会聊天、会生成内容的助手，和一个可以被托付任务、接业务系统、在关键节点再找你确认的 Agent，根本不是同一类产品。

## ReAct 为什么开始撞墙

文章把单体 ReAct 架构的问题总结得很到位，我觉得可以压缩成一句话：它太像一个边想边走、没有全局控制的大脑了。短任务还能靠能力强行完成，但一旦任务链条拉长，问题会集中爆发。

最典型的是错误级联。只要每一步都不是 100% 正确，长链任务的总成功率就会迅速塌掉。再加上缺乏明确规划阶段，系统很容易在局部看起来合理、全局却已经走偏。最后，朴素 RAG 或无时间结构的记忆系统还会让上下文不断“污染”，把过期信息和当前状态混在一起。

我觉得这也是 2025 年一个很明显的共识转向：很多团队不再把 ReAct 当作终局范式，而是把它看成一代过渡架构。它帮行业完成了从静态问答到工具调用的第一步，但在更长、更复杂、更接近真实业务的任务上，它已经越来越暴露出结构性短板。

## 从快思考到慢思考，意味着架构开始为“不确定性”预留预算

文章里关于 System 2 和 test-time compute 的部分我很认同，因为它揭示了 Agent 架构的一个深层变化：大家开始接受，不是每个问题都该被同一种推理深度处理。

过去很多系统的隐含假设是，模型拿到上下文后尽快往下走；而现在更常见的思路是，先让系统识别“当前到底有多不确定”，再决定要不要进入更昂贵的思考模式。于是你会看到 verifier、并行推理分支、MCTS、熵驱动触发器这些东西一起出现。

这背后其实是一个很实用的工程逻辑：如果问题简单，就别浪费 token；如果问题复杂，就必须显式为思考留预算。换句话说，架构不再只是在追求平均速度，而是在学着把算力分配给真正需要慢思考的节点。

这也是为什么我觉得 2025 年很多 Agent 讨论的关键词不再只是“模型更强”，而是“如何合理地让模型多想一点”。这听起来像小变化，实际上意味着系统正在从单次推理产品，转向带有推理调度能力的执行系统。

## 多 Agent、事件驱动和时序记忆，其实都在服务“可托付执行”

文章后半段谈了很多东西：多 Agent 系统、LangGraph、Swarm、MCP、A2A、Temporal Knowledge Graph、Headless Agent、AgentOps、HITL、GenUI。看上去很发散，但如果往回收，会发现它们大多在服务同一件事：**如何把 Agent 从对话框里解放出来，让它真的承担持续执行任务的责任。**

多 Agent 系统解决的是单体系统在复杂任务里顾此失彼的问题；事件驱动和状态机编排解决的是聊天式交互无法稳定承接后台流程的问题；时序记忆和知识图谱解决的是“相似不等于当前正确”这个业务系统最怕的问题。

这几条线放在一起看，会发现产品和架构是在同步演进的：

- 产品侧从聊天框转向可观测、可审批、可回滚的任务看板
- 架构侧从单体 ReAct 转向 planner、executor、verifier 的多角色协作
- 数据侧从朴素检索转向带时间约束的记忆体系

也就是说，2025 年不是某一个技术点突然爆发，而是整个 Agent 栈开始围绕“可托付执行”重写自己的默认设定。

## 商业模式也在变化，因为 Agent 不再只是提效工具

这篇综述还有一个我觉得很现实的部分，就是它把商业模式的变化也放进来了。传统 seat-based SaaS 默认假设是：软件服务的人越多，价值越大，所以按账号收费合理。但 Agent 一旦开始替代一部分具体劳动，这个前提就被削弱了。

如果一个 Agent 真的能覆盖几个人的部分工作，按 seat 收费反而会和客户价值发生错位。所以 2025 年你会看到 token-based、outcome-based、甚至 Agent-as-a-Service 这类定价方式越来越常见。这个变化本身其实说明了一件事：市场已经不再把 Agent 看成单纯的软件功能，而是开始把它看成一种接近“数字劳动力”的东西。

对产品团队来说，这不是定价部门的尾部问题，而是会反过来影响功能设计、日志颗粒度、结果归因、风控机制和交付界面。因为当你按结果收费时，产品必须更像一个可以被审计的执行系统，而不是一个“差不多挺智能”的工具。

## 我的看法：2025 年真正成熟的，不是 Agent 技术本身，而是大家对 Agent 的预期

我读完这篇文章最大的感受是，2025 年真正发生质变的，可能不是某项单独技术，而是行业终于开始更准确地理解 Agent 是什么、又不是什么。

大家开始知道，Copilot 不是终局；单体 ReAct 也不是终局；聊天框不是终局。真正可用的 Agent，需要更强的规划、更细的执行分层、更稳定的记忆结构、更明确的人类接管点，以及更接近业务结果的衡量方式。

所以如果要总结这篇文章，我会说它最有价值的地方，是把 2025 年的“热闹”翻译成了一套更冷静的判断：AI 正在从生成内容的界面层能力，走向承担业务任务的系统层能力。而一旦进入系统层，竞争就不再只发生在模型参数上，而会越来越多地发生在编排架构、记忆资产、观测能力和业务整合深度上。
