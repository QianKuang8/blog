LangChain 这篇 [How agents can use filesystems for context engineering](https://blog.langchain.com/how-agents-can-use-filesystems-for-context-engineering/) 我觉得很有启发，因为它把一个经常被当成“实现细节”的东西重新抬到了架构层：**文件系统不是 Agent 的附属工具，而正在变成 Agent 的外部记忆、状态缓冲区和上下文交换层。**

这篇文章真正想回答的问题其实很朴素：既然上下文窗口昂贵、脆弱、而且会随着任务变长越来越乱，那复杂 Agent 到底该把大量中间信息放在哪里？作者给出的答案非常直接，也非常工程化：放到文件系统里。

## 这篇文章最好的模型，是 Blue / Green / Red

文章里有一个我很喜欢的抽象：把上下文分成 Blue、Green、Red 三层。

- Blue 是环境里全部可用的信息
- Green 是当前步骤真正需要的信息
- Red 是 Agent 实际检索并塞进上下文窗口的信息

我觉得这个模型很有用，因为它把上下文工程的目标讲得非常清楚了。真正理想的状态不是“检索越多越好”，而是尽量让 Red 逼近 Green，并且不要把无关的 Blue 一股脑推给模型。

一旦用这个视角看问题，很多 Agent 的失败就很容易解释了：要么是检索太多，Red 远大于 Green，导致窗口被噪声淹没；要么是检索不到关键的利基信息，Red 根本覆盖不了 Green；要么是 Green 本身就太大，需要分阶段读取和暂存。

## 文件系统最重要的价值，是把“大上下文”从主窗口里卸载出去

文章里我最认同的一点，是把文件系统明确当作 scratchpad。很多工具调用、网页抓取、日志输出、代码检索结果，本身都可能是成千上万 token 的长文本。如果这些内容每次都直接塞进消息历史，窗口很快就会膨胀，后续推理质量也会一起掉下去。

而文件系统提供的是另一条路：先把大块内容写到外部，再让 Agent 用 `ls`、`glob`、`grep`、`read`、`read_lines` 等工具按需读取局部片段。这样上下文窗口只保留当前推理真正要消费的部分，而不是承担整份原始材料。

我觉得这是 Agent 设计里一个非常关键的变化。上下文窗口不再是唯一记忆容器，它更像是当前注意力区；真正的大体量信息则被卸载到一个可随机访问的外部存储层里。

## 为什么文件系统比纯语义检索更适合很多 Agent 场景

文章并没有否定向量检索，但它很清楚地指出，在很多技术场景里，文件系统工具有一种被低估的优势：确定性。

对代码符号、日志关键词、配置字段、API 名称这类语义稀疏的信息，向量检索未必稳定，反而 `grep`、目录遍历、按行读取这类原生文件操作更可靠。尤其在代码库和复杂文档环境里，目录结构本身就携带了大量逻辑信息，而文件系统天然保留了这种层级关系。

这也是为什么很多 coding agent 最后都在回到文件原语。不是因为它“原始”，而是因为模型本身在预训练阶段就已经学会了如何理解和操作这套 Unix 风格世界。某种程度上说，这是一种顺着模型先验能力搭系统，而不是硬塞一套完全私有的抽象。

## 文件系统还改变了长期任务和多智能体协作的组织方式

文章后面我觉得特别有意思的一点，是它不仅把文件系统当存储，还把它当协作介质。

长任务可以把计划、阶段性结论和失败记录写到文件里，后面随时再读回来，相当于主动操控模型的注意力焦点；多智能体协作时，不同子代理也可以通过共享文件交接中间产物，而不是完全依赖口头摘要式的消息传递。这样能明显减少“传声筒”问题。

文章还提到技能文件、指令文件、偏好文件的动态加载。我很认同这种方向，因为它意味着系统指令不一定要永远硬编码在 system prompt 里，而可以变成一组按需读取、可以持续更新的外部资产。

## 我的看法：文件系统会越来越像 Agent 的默认内存层

我读完这篇文章最大的感受是，文件系统之所以重要，不是因为它比别的方案“更高级”，恰恰相反，是因为它足够朴素、足够稳定、足够通用。

它既能存大块原始材料，也能存中间计划、运行日志、技能说明和共享状态；既适合单 Agent 长轨迹，也适合多 Agent 协作；既能和向量检索结合，也能单独承担很多精确检索任务。更重要的是，它把上下文窗口从“唯一记忆载体”降格成了“当前注意力窗口”，这是我认为复杂 Agent 走向长期运行所必须经历的一步。

所以如果要总结这篇文章，我会说：它真正说明的，不是“文件系统很好用”，而是**在 Agent 时代，文件系统很可能会成为最默认的外部记忆层。**
