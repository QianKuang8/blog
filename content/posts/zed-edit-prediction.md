Zed 这篇 [Edit Prediction](https://zed.dev/blog/edit-prediction) 我觉得特别值得读，因为它不是在讲“我们又做了一个更强的补全模型”，而是在重新定义编辑器里 AI 应该解决什么问题。传统补全默认的问题是：光标后面下一段文本该写什么？而 Zed 提出的其实是另一个问题：**开发者刚刚这一串操作，下一步最可能会改哪里、怎么改？**

这看起来像是从 completion 到 next edit 的一次小升级，实际上是完全不同的交互范式。因为一旦任务目标从“补文本”变成“预测编辑”，模型、训练数据、评估方法，甚至 Tab 键的交互逻辑，都会跟着一起变化。

## 真正被挑战的，不是补全质量，而是补全的任务定义

Zed 文章里最核心的判断，是 Fill-In-The-Middle 这种经典补全范式已经不够覆盖真实开发行为了。FIM 很适合处理“我在当前光标位置还要继续写下去”这种问题，但它默认代码编辑是线性的，默认光标就是全部意图来源。

可现实开发并不是这样。很多时候你做的是重构、参数调整、结构性替换，甚至只是刚改了一处定义，接下来还要在同一片段里的别处补齐连锁修改。问题不在“能不能生成几行代码”，而在“能不能看懂这是一段正在发生的编辑过程”。

所以 Zed 真正做的，不是把补全再做快一点，而是把任务从“字符串填空”换成了“编辑预测”。这一步很关键，因为它意味着 IDE 里的 AI 开始从文本生成器转向编辑协作者。

## Zed 的核心选择，是用 Rewriting 取代细粒度 diff

文章里我觉得最有意思的技术选择，是 **Editing by Rewriting**。Zed 没有让模型去学那种特别细碎的补丁操作，而是把光标附近的一段代码当成一个可重写块，让模型直接输出重写后的版本。

这听起来有点“笨重”，因为重写看起来比 diff 更费 token，但它背后的直觉其实很合理：模型未必擅长在非常细颗粒度的编辑格式里稳定操作，却往往更擅长重写一段结构完整的代码。与其逼模型像文本编辑器那样做微操作，不如让它在自己更擅长的表达方式里完成预测。

我很认同这个思路。因为很多编辑任务并不是简单的增删几字符，而是一个局部结构重新组织的问题。只要任务进入结构变换层面，重写往往比细粒度 patch 更符合模型的天然能力边界。

## 这套方案真正难的地方，是如何把“重写”做得足够快

当然，Rewriting 的代价也非常明显：它天生比 FIM 需要生成更多 token。如果这一点处理不好，体验再聪明也会被延迟拖死。Zed 也非常清楚这一点，所以文章后半部分几乎都在围绕速度优化展开。

最关键的优化是 speculative decoding。因为 edit prediction 的输出和输入大部分内容其实是相同的，真正变化只集中在少数位置，所以系统可以把原始片段当成强先验，借助 n-gram 匹配去寻找可以并行生成的跳跃点。

这里我觉得 Zed 的工程思路非常清楚：它没有因为“模型结构更自然”就忽略现实约束，而是反过来承认，如果不能把 p50 压到 200ms 左右，这个产品在编辑器里就不成立。换句话说，交互范式的升级必须和延迟预算一起设计，而不是先做对、再想办法跑快。

## 训练流程说明了一个现实：prompt engineering 很快会撞墙

文章里还有一点我很喜欢，就是它对 prompt engineering 的态度很诚实。Zed 最开始也尝试过直接拿大模型配系统提示去做，前几组 eval 看起来还不错，但一旦用例变多，很快就进入了“修一个坏另一个”的循环。

这其实很典型。只要任务本身已经脱离简单补全，进入结构化编辑预测，你就很难靠 prompt 把所有行为稳定住。Zed 后来转向 SFT，再用 DPO 修那些随机插删的坏 case，本质上是在承认：**这已经不是 prompt 打磨问题，而是一个需要专门训练的数据问题。**

我觉得这点很重要。因为它说明 next-edit 不只是一个 UI feature，而是一个足够独立、足够复杂、值得单独建模和训练的新任务。它和“给现成代码模型再套一层提示”不是一个工程层级。

## LLM-as-a-Judge 的选择，说明评估目标已经变了

Zed 用 Claude 当评判者，而不是做严格字符串断言，这一点也很有代表性。因为当任务从补全变成编辑预测之后，“和参考答案逐 token 一样”已经不再是最重要的事情。更重要的是，模型给出的重写是否真正实现了那个编辑意图。

这背后其实是评估标准的转向：从文本一致性走向功能和意图一致性。只要你接受这一点，就会发现很多传统测试方法在这类任务上天然很脆弱。模型可能和 ground truth 不完全一致，但依然是一个优秀的编辑建议。

这也再次说明，Next Edit 不是补全的延长线，而是一个需要重新定义评估方式的新任务。

## 我的看法：Zed 真正领先的，是它把“编辑预测”做成了编辑器原生体验

我读完之后最大的感受是，Zed 的优势不只在模型开源或者训练细节，而在于它从一开始就把这件事当成一个编辑器体验问题来设计，而不是单纯的模型能力展示。

比如 `tab` 如何和 LSP completion 共存，什么时候需要 `option/alt` 预览，Linux 下默认快捷键如何处理，这些看起来是很小的产品细节，但恰恰说明 Zed 真正在意的是：这个能力是否会破坏开发者原有的编辑肌肉记忆。

所以如果要总结这篇文章，我会说它最重要的地方不是“Zeta 是一个开源 7B 模型”，而是它证明了一件事：**编辑器里的 AI 下一阶段，不是继续把补全做得更长，而是要开始理解编辑本身。** 而在这条路上，真正决定体验成败的，不只是哪家模型更强，还包括谁更懂编辑器原生交互应该怎么被改造。
