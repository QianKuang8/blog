Cursor 这篇 [Dynamic context discovery](https://cursor.com/cn/blog/dynamic-context-discovery) 我觉得非常值得读，因为它把一个很多人隐约已经感觉到、但还没有被讲得足够清楚的趋势直接点破了：**随着模型作为 Agent 的能力变强，系统不一定要在一开始就塞给它更多上下文，很多时候更好的做法反而是给得更少，然后让它自己去找。**

这篇文章最有价值的地方，在于它不是泛泛地说“上下文工程很重要”，而是明确挑战了静态上下文的默认思路。过去我们经常觉得，为了避免模型遗漏细节，最好预先把尽可能多的背景、历史和工具信息塞进去；但 Cursor 的经验是，这样做反而会让系统越来越重、越来越乱、越来越贵。

## 这篇文章真正反对的，是“预先灌满上下文窗口”

文章开头对静态上下文的问题总结得很到位。首先当然是 token 成本和长轨迹下的质量退化；但更重要的是，一旦系统习惯于在前面预埋大量细节，这些细节本身就会逐渐变成噪声，甚至互相冲突。

另一种常见补救办法是摘要和截断，但文章指出这其实都是有损压缩。你当然能把历史压短，但压缩之后，Agent 对任务的理解也可能跟着一起变形。一旦关键状态、失败证据或者某个具体约束在摘要里丢了，后面的推理就会建立在一个退化版本的世界上。

所以 Cursor 的核心判断很明确：问题不是“怎么把更多东西塞进窗口”，而是“怎么让 Agent 在需要的时候自己拿到正确的东西”。

## File as primitive，是这篇文章最核心的工程选择

文章里我最认同的一点，是它没有为这个问题再造一层特别复杂的私有元数据系统，而是回到了一个非常简单的原语：文件。

Cursor 选择把长工具响应、对话历史、技能定义、终端输出、MCP 工具描述这些内容都映射到文件系统中。这样做的意义不只是存储，而是把上下文访问从线性注入变成了随机访问。Agent 不必在一开始拥有所有细节，而可以通过 `grep`、文件读取、局部引用等方式按需获取。

我觉得这个设计很重要，因为它说明“动态上下文发现”不是一个抽象理念，而是一种很具体的运行时结构：把信息留在外部，让 Agent 在任务推进中主动检索，而不是由系统一次性灌输。

## 文章展示的几种机制，其实都在服务同一个原则

Cursor 文中举了不少具体机制，我觉得可以统一理解成一句话：**尽量把大块、低频、可延迟的信息挪出主窗口，只在需要时再读进来。**

比如：

- 长工具响应不直接塞回会话，而是写到文件里，只返回句柄
- 对话历史被压缩后，完整轨迹仍然保留为可回溯文件
- Skills 不是全部预加载，而是只给名称和简介，需要时再读正文
- MCP 工具按服务器分组，而不是扁平铺开，让模型更容易把它们当作逻辑单元理解
- 长生命周期终端输出持续同步到文件，方便后续定向检索

这些机制表面不同，背后其实都在做同一件事：把 Agent 从被动接收上下文，变成主动发现上下文。

## 反直觉但关键的一点：更强的模型，反而更适合“少给一点”

我觉得这篇文章最有意思的地方，就是它的结论非常反直觉。很多人会以为模型变强以后，系统就更应该一股脑给更多背景；但 Cursor 的观察恰好相反，正因为模型更能理解工具和环境，它才更适合自己去发现相关上下文。

这意味着未来 Agent harness 的重点会逐渐变化。过去你可能更关注“怎么提前准备一个尽可能完整的超级 prompt”；以后更重要的可能是“怎么给 Agent 一组足够可靠的发现工具、一套足够清楚的检索习惯，以及一个足够简单稳定的外部信息层”。

文章提到在 MCP 工具调用场景下，总 token 消耗下降了 46.9%。我觉得这个数字最值得看的地方，不只是省钱，而是它说明系统通过减少上下文噪声，反而提高了上下文利用效率。

## 我的看法：编程 Agent 的下一步，不是更会背上下文，而是更会找上下文

我读完之后最大的感受是，动态上下文发现很可能会成为 coding agent 的下一阶段默认范式。因为代码库、日志、终端输出、工具说明、本地规则这些东西，本来就不是一份适合整体朗读给模型听的材料，它们更像一个需要被探索的环境。

所以如果要总结这篇文章，我会说：它真正提出的，不是一个新优化技巧，而是一种新的协作假设。系统不用再假设自己能提前知道什么信息最重要，而是把更多发现权交给 Agent 本身。**从“预先塞满”到“按需发现”，这可能会是编程 Agent 架构变化里非常关键的一步。**
