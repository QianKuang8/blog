Augment 这两篇 [The AI research behind Next Edit](https://www.augmentcode.com/blog/the-ai-research-behind-next-edit) 和 [Introducing Next Edit for VS Code](https://www.augmentcode.com/blog/introducing-next-edit-for-vscode) 我觉得最值得看的地方，不是它也做了 Next Edit，而是它把这个问题拆得非常工程化：不是问“模型能不能猜中下一步”，而是分成三个独立问题来解，分别是用户到底在做什么、这些变化应该落到哪里、最后又该怎样高效地把改动执行出来。

这种拆法很重要，因为它把“编辑预测”从一个模糊的产品能力，落成了一个完整的系统问题。尤其当目标从单文件局部编辑升级到整个 workspace 的连锁传播时，你会发现真正困难的不是生成几行代码，而是如何把一处修改背后的涟漪效应稳定地找出来。

## Augment 真正抓住的问题，是跨文件编辑的“涟漪效应”

文章举的 `session_id` 例子其实非常典型。开发者在一个类里加了一个字段，看起来只是局部改动，但它很可能会波及 SQL 查询、类型定义、序列化结构、测试用例和别处的业务逻辑。真正费时间的部分，往往不是原始修改，而是这些后续传播。

这也是为什么我觉得 Augment 的问题定义比很多“下一步补全”更贴近真实开发。它不是在讨论如何更聪明地补一个局部 token，而是在处理一类更现实的负担：开发者明明已经知道自己要改什么，但 workspace 太大，依赖链太散，传播成本远高于原始意图本身。

从这个角度看，Next Edit 在 Augment 这里更像一个“变化传播引擎”，而不是传统意义上的补全能力。

## 三段式拆分，是这套方案最有工程感的地方

Augment 把问题拆成了 What、Where、How 三层。我觉得这是全文最关键的地方，因为它说明真正成熟的 IDE 智能能力，不应该把所有问题都丢给一个模型端到端完成。

- **What**：从复杂、非线性的编辑历史里推断开发者真正意图
- **Where**：在整个 workspace 中找出哪些位置需要跟进修改
- **How**：把这些修改以足够快、足够稳的方式真正生成出来

这个拆法的好处是，每一层都可以按自己的优化目标去设计。意图推断更像行为理解问题，定位更像检索和依赖发现问题，执行则更像生成和应用问题。与其指望一个通用大模型同时把这三件事都做好，不如把系统拆开，让每层各自承担最适合它的任务。

我觉得这就是 Augment 比很多“Agent 式遍历代码库”方案更现实的地方。它没有把一切都交给一个会思考的通用体，而是愿意承认，不同阶段其实更适合不同的机制。

## 真正拉开差距的，是定位层而不是生成层

文章里我最认同的一点，是它没有迷信用大模型去做代码导航。Augment 明确把定位问题交给了专门训练的 retriever，而不是让 LLM 像 OpenDevin 那样一点点翻目录、读文件、猜依赖。

这背后其实是一个非常务实的判断：如果你想在 IDE 里做实时、后台、workspace 级联想，靠大模型慢慢导航根本不现实。它太慢，也太浪费上下文。真正可用的方案必须把“Where”单独拿出来，做成一个可扩展、可低延迟运行的定位层。

我觉得这是 Next Edit 走向实用化的关键一步。很多 AI 编码产品最后卡住，不是因为模型不会生成，而是因为它们找不到正确位置，或者找位置的成本高得让交互无法成立。Augment 至少把这个瓶颈看得很准。

## 为了避免“撤销用户工作”，训练数据必须保留中间态

文章里提到的 undo bias，我觉得非常有启发。因为很多训练集只看初始状态和最终状态，这会让模型学到一种危险直觉：中间修改只是噪声，目标是尽快回到某个“稳定终态”。

但真实开发不是这样。开发者最新那一步修改，往往正是意图最强的信号。模型如果老想着把局部变化“纠正回去”，就会表现得像一个自作聪明的工具，甚至直接破坏用户工作。

Augment 之所以强调保留中间编辑状态，本质上是在修正模型对“编辑过程”的理解：编辑不是从 A 到 B 的静态差异，而是一串具有时间顺序和意图浓度的动作链。这个问题如果训练时不处理，产品层再怎么包装都会很别扭。

这一点也说明，Next Edit 的关键数据不是 commit history，而是 interaction history。

## 它最后落到的，其实是一套更像基础设施的执行层

在 “How” 这一层，Augment 也没有走全文件重写路线，而是设计了更紧凑的 diff 解码格式。原因很现实：如果每次都重写整段甚至整文件，延迟、token 消耗和错误率都会迅速上升，尤其在大文件里很容易遇到注意力瓶颈。

所以它的执行层重点不是“让模型写得更华丽”，而是让模型只生成必要的 delta，而且这个 delta 还能被无歧义地应用回原始代码。这种设计很像是在告诉我们：真正能进入日常工作流的 AI 编辑能力，最终一定会越来越像基础设施，而不是像一次大模型即兴表演。

我很认同这一点。因为只要产品目标是“后台、无感、连续”，系统就必须在生成层尽量克制，把复杂度转移到检索、定位和应用机制上。

## 我的看法：未来 IDE 的竞争，越来越像在比谁更会传播修改

我读完 Augment 这两篇文章最大的感受是，未来编辑器里的 AI 竞争，可能不再只是“谁的聊天更聪明”或者“谁的补全更像人”，而会越来越像在比谁更会处理 workspace 级变化传播。

因为真实开发里最耗人精力的，往往不是写第一行代码，而是知道一处变化会波及哪里、自己还漏了什么、哪些地方应该同步改掉。谁能把这种传播成本系统性地压下去，谁就更接近真正改变开发体验。

所以如果要总结 Augment 的价值，我会说它最重要的贡献，不是又做出了一个 Next Edit 功能，而是把这件事明确地重构成一个三层系统：意图理解、全局定位、低延迟执行。这个框架本身，比单次 demo 表现更值得记住。
