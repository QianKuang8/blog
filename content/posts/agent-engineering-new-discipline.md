LangChain 这篇 [Agent Engineering: A New Discipline](https://blog.langchain.com/agent-engineering-a-new-discipline/) 我觉得很值得读，因为它点破了一个经常被忽略的问题：很多团队以为自己是在“做一个更复杂的 LLM 应用”，实际上他们已经在处理一种和传统软件很不一样的系统。系统一旦会推理、会调用工具、会根据上下文自我调整，老一套软件工程手册就开始失灵。

所以这篇文章最重要的地方，不是提出了一个新名词，而是给这种失灵感找到了一个准确的工程语言。所谓 Agent Engineering，本质上是在说：面对非确定性系统，团队必须重新组织自己的方法论，尤其是如何测试、如何上线、如何根据生产数据迭代。

## 真正的问题不是“模型不够强”，而是系统不再是确定性的

文章开篇讲得很直接：从“本地能跑”到“生产可靠”之间的距离，在 Agent 系统里会突然变得极大。原因不是工程师不够努力，而是系统本身的性质变了。

传统软件默认输入空间相对可控，流程可以预先穷举，异常可以被显式编码。但 Agent 不一样。用户输入是自然语言，工具调用链条是动态展开的，模型推理过程又天然带有概率性。于是你很快会发现，过去很多熟悉的工程直觉开始变钝：

- 没有真正意义上的“标准输入”
- 逻辑不再完全写在代码里，而是部分沉到 prompt 和模型行为中
- 系统“在线”不等于系统“正确”

这也是为什么很多 Agent 产品看起来没宕机、日志也没爆炸，但用户体验仍然一团糟。它们不是传统意义上的 outage，而是行为层面的偏航。LangChain 这篇文章最有价值的一点，就是它把这种偏航提升成了第一类工程问题。

## Agent Engineering 真正要求团队改变什么

文章把 Agent Engineering 定义成一个循环：**build, test, ship, observe, refine**。这看起来像一句很正常的话，但我觉得它真正想推动的是一种组织习惯的改变。

在传统软件里，很多团队习惯把“上线”当作验证完成之后的终点；在 Agent 系统里，上线更像是进入了一个新的观测阶段。因为你不可能在预发布环境里覆盖真实世界的输入多样性，很多关键问题只能靠生产 trace 暴露出来。

这意味着团队的工作重心会变：

- prompt 不再只是临时配置，而变成需要被持续维护的系统逻辑
- 工具定义不再只是接口包装，而是 Agent 行为边界的一部分
- 生产 trace 不再只是排障材料，而是后续评估和回归测试的核心资产

换句话说，Agent Engineering 并不是“模型时代的软件工程”这么简单，它更像是把 prompt、工具、评估和生产观测一起纳入工程主循环的一套新实践。

## 它为什么一定是跨职能工作，而不是单纯的工程问题

LangChain 文中把 Agent Engineering 拆成三个能力维度：产品思维、工程能力和数据科学。我觉得这个拆法是对的，因为它说明 Agent 系统很难再完全由单一职能独立交付。

产品侧要定义“任务到底完成成什么样才算好”，也就是 job to be done 和评估口径；工程侧要提供 durable execution、HITL、记忆管理、工具集成这些基础设施；数据侧则必须接手评估、监控和错误分析，把系统行为从模糊体验变成可观察指标。

我很认同这背后的含义：Agent 产品不是“前端提需求、后端去实现、上线后看报表”的那种流水线模式。它更像一个需要共同塑形的非确定性系统。谁来写 prompt、谁来定义工具、谁来判断某次失败是模型问题还是流程问题，这些边界都会变得模糊，但也因此更需要协作。

从这个角度看，Agent Engineering 确实更像一种新学科，而不只是某个岗位的技能升级。

## “Ship to Learn” 不是口号，而是被系统性质逼出来的

整篇文章里我最认同的一点，是它对 **Ship-to-Learn** 的强调。很多人听到这句话会本能地警惕，觉得是不是在为“先上线再说”找借口。但放在 Agent 系统里，它并不是偷懒，而是现实。

因为只要输入是自然语言，就不可能像传统流程一样把所有用户行为预先枚举出来。你可以测试一批典型案例，修掉一批明显错误，但永远会有新的边缘情况在真实生产里出现。真正的问题从来不是“能不能提前全测出来”，而是“上线之后能不能快速看见、快速定位、快速修正”。

这就要求系统必须具备几种能力：

- 能追踪每次交互和每次工具调用的上下文
- 能把真实失败 case 回灌到评估集里
- 能让 prompt 和工具逻辑在天级别迭代，而不是季度级别迭代

如果没有这些能力，Ship-to-Learn 就会沦为放任系统在生产中乱撞；但如果这些基础设施都在，那它反而是最现实的工程策略。

## 这篇文章真正抬高的是“观测能力”的地位

文章里提到 tracing 的重要性，我觉得这一点应该被更认真地看待。因为在 Agent 系统里，很多问题你不是靠看报错日志就能发现，而是要看它在当时拿到了什么上下文、为什么调用了这个工具、为什么在这个节点做出了这个判断。

这意味着 tracing 不是一个附属观测功能，而是 Agent 工程最核心的调试入口。没有足够细粒度的 trace，团队几乎不可能真正理解系统为什么表现失常，更别提稳定迭代。

这也是为什么我会把这篇文章和 Anthropic 那篇《Building Effective Agents》放在一起看。前者更强调工程组织和迭代纪律，后者更强调架构模式和工具接口设计。两者其实在说同一件事：Agent 不是“跑起来就行”的系统，它必须被设计成一个可以持续被理解和修正的系统。

## 我的看法：Agent Engineering 的核心，不是造新概念，而是承认旧方法已经不够

我读完这篇文章后的最大感受是，它真正有价值的地方不是创造了一个听起来更酷的 label，而是逼团队承认一件事：当系统开始表现出非确定性、多步骤、自主决策这些特征时，传统软件工程并没有一套现成答案可以直接套用。

所以 Agent Engineering 最重要的贡献，不是把 Agent 讲得更神秘，恰恰相反，它是在帮助大家把这种新系统重新拉回到工程语境里。你需要评估，需要 trace，需要持续回归，需要基于真实生产行为去修 prompt 和工具，而不是幻想一次性把系统设计完。

如果要用一句话总结这篇文章，我会说：它在提醒我们，Agent 的难点从来不只是模型能力，而是你有没有能力把一个不可穷举、不可完全预测的系统，逐步驯化成一个可靠的生产系统。
